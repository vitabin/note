## 1. 기초 이론: 펄스(Pulse)와 데이터 변환

소프트웨어의 논리값(0과 1)이 물리적인 하드웨어에서 어떻게 표현되고 전달되는지에 대한 근본적인 이해가 필요합니다.

### 1.1. 펄스(Pulse)의 물리적 정의

전기/전자 공학에서 펄스란 **전압이나 전류가 기본 상태(Baseline)에서 일시적으로 다른 상태로 급격히 변했다가 다시 원래의 상태로 돌아오는 현상**을 의미합니다.

디지털 제어에서 다루는 펄스(사각파, Square Wave)의 핵심 파라미터는 다음과 같습니다.

- **진폭 (Amplitude / Voltage Level):** 펄스의 높이. 신호가 0V(Low)에서 몇 V(High)까지 올라가는가를 나타냅니다. (예: STM32의 기본 논리 레벨은 3.3V)
    
- **상승 엣지 (Rising Edge) / 하강 엣지 (Falling Edge):** 신호가 Low에서 High로 변하는 순간(상승)과 High에서 Low로 변하는 순간(하강)입니다. 타이머 하드웨어는 이 '변화하는 순간'을 감지하여 카운트를 올립니다.
    
- **펄스 폭 (Pulse Width):** 신호가 High 상태를 유지하는 시간입니다.
    
- **주파수 (Frequency):** 1초에 펄스가 몇 번 반복되는지(Hz)를 나타내며, 이 프로젝트에서는 모터의 회전 속도에 비례하여 펄스 주파수가 높아집니다.
    

### 1.2. 로터리 인코더의 펄스 생성 원리

서보 모터에 부착된 광학식 로터리 인코더는 물리적인 회전을 전기적 펄스로 변환하는 센서입니다.

- **생성 메커니즘:** 모터 축에 연결된 슬릿(구멍이 뚫린 원판)이 회전하면서 발광부(LED)의 빛을 통과시키거나 차단합니다. 수광부(포토 다이오드)는 빛을 받으면 전압을 올리고(High), 차단되면 전압을 내립니다(Low). 이것이 펄스의 연속으로 나타납니다.
    
- **A/B상 (Quadrature Phase)과 방향 판별:** 회전량(펄스 개수)뿐만 아니라 회전 방향을 알기 위해 위상이 90도 어긋난 두 개의 펄스(A상, B상)를 출력합니다. A상이 먼저 High가 되는지, B상이 먼저 High가 되는지를 하드웨어로 비교하여 정/역방향을 판별합니다.
    

### 1.3. 차동 신호 (Differential Signal)와 노이즈 상쇄

산업 현장의 극심한 전기적 노이즈를 극복하기 위해 서보 모터는 펄스를 보낼 때 단일 선이 아닌 차동 신호(RS-422 등) 규격을 사용합니다.

- **원리:** 하나의 신호(A)를 보낼 때, 위상이 완전히 반대인 역신호(A-bar)를 두 가닥의 전선으로 동시에 보냅니다.
    
- **노이즈 제거:** 전송 과정에서 외부 노이즈가 유입되면 두 선에 동일한 크기의 노이즈가 더해집니다. 수신부(리시버 IC)에서 두 선의 전압 '차이'를 계산하면($(A + Noise) - (\bar{A} + Noise)$), 노이즈는 수학적으로 완전히 소거되고 원본 신호만 증폭되어 MCU로 전달됩니다.

## 2. 펌웨어 제어를 위한 필수 사전 지식 (Prerequisite Knowledge)

이 프로젝트를 성공적으로 수행하기 위해 추가로 학습해야 할 키워드와 개념들입니다.

### 2.1. 임베디드 아키텍처 (Embedded Architecture)

- **메모리 맵드 I/O (Memory-mapped I/O):** MCU는 특정 메모리 주소(예: `0x40020000`)에 값을 읽고 쓰는 행위만으로 물리적인 핀의 전압을 제어하거나 하드웨어 타이머를 설정합니다. 레지스터(Register) 제어의 개념을 이해해야 합니다.
    
- **인터럽트 (Interrupt) 및 NVIC:** CPU가 `while` 문을 도는 대신, 특정 하드웨어 이벤트(예: 100us 타이머 만료, UART 수신 완료)가 발생했을 때만 실행 흐름을 멈추고 지정된 함수(ISR)를 실행하는 구조입니다. (Event-driven 아키텍처와 유사)
    
- **DMA (Direct Memory Access):** CPU의 개입 없이 주변 장치(UART 등)와 메모리 간에 데이터를 직접 전송하는 하드웨어 컨트롤러입니다. CPU 오버헤드 없이 데이터를 메모리에 적재할 때 필수적입니다.
    
- **메모리 계층 구조 (ITCM, DTCM, D-Cache):** 고성능 MCU(H7)에서 캐시 미스(Cache Miss)로 인한 실행 지연을 막기 위해 링커 스크립트를 조작하여 메모리를 명시적으로 할당하는 방법입니다.
    

### 2.2. C 언어 심화 (Advanced C Programming for Firmware)

- **비트 연산 (Bitwise Operations):** 레지스터의 특정 비트(예: 3번 비트만 1로 셋팅)만 조작하기 위해 `&`, `|`, `~`, `<<`, `>>` 연산을 자유롭게 다루어야 합니다.
    
- `volatile` **키워드:** 컴파일러 최적화를 방지하는 키워드입니다. 인터럽트나 DMA에 의해 백그라운드에서 값이 변경될 수 있는 변수(예: 상태 플래그, 하드웨어 레지스터) 선언 시 반드시 필요합니다.
    
- **구조체 패딩 (Struct Padding) 방지:** UART로 바이트 단위 데이터를 수신하여 구조체에 매핑할 때, 컴파일러가 임의로 삽입하는 패딩 바이트를 막기 위해 `#pragma pack(1)` 또는 `__attribute__((packed))`를 사용하는 법을 알아야 합니다.

## 3. 확정된 시스템 아키텍처 및 구현 방안

### 3.1. 데이터 수집 (Data Ingestion): Zero-CPU Overhead

- **방식:** 6개 서보 모터의 A/B상 펄스를 STM32 H7의 **하드웨어 타이머(Encoder Interface Mode)** 에 직결.
    
- **효과:** 하드웨어가 펄스의 엣지를 감지하고 방향을 판별하여 타이머 카운터 레지스터(`TIMx->CNT`)를 자동 갱신. CPU 개입 없이 실시간 위치 데이터 확보.
    

### 3.2. 룩업테이블(LUT) 동기화: Lock-Free 아키텍처

- **통신:** 메인 MCU로부터 UART를 통해 수신.
    
- **수신 처리 (DMA + Double Buffering):** UART 수신을 DMA로 처리하여 메모리에 직접 기록. 수신 완료(Idle Line) 인터럽트 발생 시 메모리 포인터만 스위칭하여 원자적(Atomic)으로 테이블 갱신.
    
- **캐시 일관성 관리:** L1 D-Cache 사용 시, DMA 수신 버퍼 영역은 MPU를 통해 Non-cacheable로 설정.
    

### 3.3. 매칭 알고리즘 최적화 (Write-Path & Read-Path 분리)

- **Write-Path (수신 시점):** LUT 데이터(10x128) 갱신 시, 실시간 연산 부하를 없애기 위해 허용 오차($\pm10$)를 미리 계산하여 `Min/Max` 범위를 갖는 구조체 배열로 재가공 후 저장.
    
- **Read-Path (실시간 매칭):**
    
    - **주기:** 100us(0.1ms) 주기의 하드웨어 타이머 인터럽트 구동.
        
    - **탐색 알고리즘:** 데이터가 작으므로(128행) **선형 탐색(Linear Search)** 이 가장 빠름.
        
    - **Early Exit:** 6개 축 조건 검사 시 첫 번째 축이 범위를 벗어나면 즉시 `continue` 하여 다음 행으로 이동.
        

### 3.4. 메모리 및 출력 최적화

- **TCM 활용:** Zero Wait-State를 위해 `active_lut` 배열과 연산 변수들은 **DTCM**에 배치. 타이머 ISR 코드는 **ITCM**에 배치.
    
- **트리거 출력:** 조건 만족 시 GPIO 레지스터(`BSRR`)에 직접 접근하여 최소 지연 시간으로 펄스 출력.

## 4. 현업 확인 필요 사항 (질문 리스트)

1. **입력단 (Encoder to Board):**
    
    - 서보 모터의 인코더 출력 신호 타입이 구체적으로 무엇입니까? (예: Line Driver / RS-422, Open Collector 등)
        
    - 보드 입력단에 위 신호를 3.3V 로직으로 변환해 주는 리시버 IC(예: AM26LS32) 회로가 설계되어 있습니까?
        
2. **출력단 (Board to Target):**
    
    - 우리가 발생시킬 트리거 펄스를 최종적으로 수신하는 장비는 무엇이며, 요구하는 신호 스펙(전압 레벨, Push-Pull/Open-Drain 여부)은 어떻게 됩니까?
        
    - 요구되는 트리거 펄스의 최소 유지 시간(Pulse Width)은 몇 us / ms 입니까?
        
3. **통신단 (Main MCU to Board):**
    
    - UART(RS232) 통신의 Baudrate 설정값과 LUT 데이터의 전송 프레임 규격(헤더, 테일, 체크섬 구조 등)을 제공해 주실 수 있습니까?
        
4. **핀맵 (Pin Map):**
    
    - 6개의 인코더 입력(총 12가닥)이 STM32 H7의 어느 타이머 채널(핀 번호)에 매핑되어 있습니까?

## 5. 추가 검토 필요 사항

- **시스템 신뢰성 (Reliability)**
    
    - **노이즈 대책:** 하드웨어 타이머의 디지털 필터(Digital Filter) 설정값 도출.
        
    - **예외 처리:** 비정상적인 카운트 스파이크(Spike) 발생 시 처리 로직.
        
    - **페일세이프 (Fail-safe):** UART 통신 단절 시 기존 LUT 유지 또는 안전 정지(Halt) 여부 결정.
        
- **펌웨어 초기화 및 보정 (Calibration)**
    
    - **원점 복귀 (Homing):** 장비 부팅 시 인코더 카운터를 절대 위치 0과 동기화하는 절차 (Z상 펄스 또는 리미트 스위치 활용 방안 모색).
        
- **디버깅 및 모니터링 (Monitoring)**
    
    - 로깅을 위한 별도의 UART 디버그 채널 확보 및 PC단 모니터링(예: Grafana 연동용 로컬 프록시) 구축 가능성 검토.
